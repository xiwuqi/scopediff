# ScopeDiff v0.1.1 Plan

Status: draft for maintainer review.

Target: small follow-up release after v0.1.0 launch. No new major capability, no new platform automation, and no change to ScopeDiff's positioning as a local review aid.

## Planning Inputs

- X/Twitter and LinkedIn launch posts are public, but social metrics are still placeholders.
- Reddit r/opensource was submitted and then removed by AutoModerator due account-age rules.
- GitHub currently has 5 maintainer-created issues and 1 star at the time of planning.
- GitHub traffic and npm download data are not yet useful.
- There is no confirmed external product feedback yet, so v0.1.1 should optimize for first-time clarity rather than reacting to noise.

## Release Goal

Make ScopeDiff easier for a cold maintainer to evaluate in under 5 minutes:

1. understand what the tool checks
2. run the npm package
3. inspect a realistic report
4. understand why a finding may be conservative or noisy
5. know how to report a sanitized false positive

## Non-Goals

- No SARIF output in v0.1.1.
- No automatic PR comments.
- No GitHub App or hosted service.
- No remote package reputation checks.
- No additional social launch automation.
- No claims that ScopeDiff is a complete security scanner.

## Proposed Scope

| Priority | Work Item | Linked Issue | Why Now | Acceptance Criteria |
| --- | --- | --- | --- | --- |
| P0 | Improve README example report and quick-start flow | #1 | Launch traffic lands on README first. The report should be instantly legible. | README shows the npm quick start, a realistic short finding example, and a clear "review aid, not full audit" boundary. Commands are verified. |
| P0 | Add common false-positive guidance | #3 | Launch copy invites false-positive feedback. Users need a safe reporting path. | New docs page explains common false positives, sanitized reporting, and what not to include. README links to it. |
| P1 | Add another MCP config fixture shape | #2 | Real repositories will vary in MCP config layout. One more fixture improves confidence without expanding scope. | A sanitized fixture is added, covered by tests, and documented as supported. |
| P1 | Add more GitHub Actions permission expansion cases | #4 | Workflow permission changes are a strong part of the value proposition. | Add focused fixture tests for permission expansion cases without broad workflow parsing rewrites. |
| P2 | Tighten finding copy where it reduces confusion | Related to #1/#3 | Conservative findings need careful wording. | Findings avoid implying exploitability and keep suggested review actions concrete. |

## Deferred

| Item | Reason |
| --- | --- |
| SARIF output (#5) | Useful, but it changes integration surface and deserves a separate design pass. |
| PR comments | Requires permissions and public side effects. Keep opt-in and post-MVP. |
| Product Hunt / more Reddit | Initial data is not strong enough; improve project clarity first. |
| Remote registry/package reputation | Requires network policy, caching, and trust-model design. |

## Suggested File Changes

Documentation:

- `README.md`
- `docs/common-false-positives.md`
- `docs/demo/scopediff-report.md`
- `docs/demo/demo-scenario.md`

Tests and fixtures:

- `tests/fixtures/**`
- `tests/**/*.test.ts`

Possible source copy-only updates:

- `src/rules/**`
- `src/renderers/**`

Avoid broad refactors unless a test exposes a real bug.

## Implementation Order

1. Update README quick-start and example report.
2. Add `docs/common-false-positives.md` and link it from README.
3. Add one MCP config fixture and test.
4. Add GitHub Actions permission expansion fixtures and tests.
5. Review finding wording for overstatement.
6. Run `npm run check`.
7. Run package smoke commands:
   - `npx scopediff@latest --help`
   - local build scan/diff against fixtures
8. Prepare `docs/release-notes-v0.1.1.md`.

## Verification Checklist

- `npm run check` passes.
- README commands are real and current.
- New docs do not include private repo data, real secrets, or user data.
- New fixtures are synthetic and clearly safe.
- Markdown report output stays stable.
- JSON report output stays stable.
- No telemetry, network upload, or token storage is introduced.
- No lifecycle scripts are added.

## Release Criteria

v0.1.1 is ready when:

- #1 and #3 are fully addressed.
- At least one of #2 or #4 is addressed with tests.
- Release notes explain what changed and what did not.
- No public claims are updated beyond the actual implemented changes.

## Success Metrics After v0.1.1

Use signals, not vanity:

- More README path views or npm package views after release.
- At least one real user question, issue, or discussion comment.
- Fewer confused reactions about ScopeDiff being a scanner versus a review aid.
- Higher-quality false-positive reports with sanitized examples.
- npm downloads become available and show real installs.
